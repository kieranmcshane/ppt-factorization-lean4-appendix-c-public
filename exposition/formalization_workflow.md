The checked right gap-three reducer consumes the branch `pi(b,c)` by
transitivity with the crossing link `pi(a,c)`.  The adjacent-repeat branch
`S(b+1,c)` has been normalized to the fixed-point branch `pi(b+1)=b+1`,

The branch `pi(b+1,c)` is now routed through the shifted crossing
`(a,b,b+1,d)`: it gives `pi(a,b+1)`, the large-gap right input gives
`S(b+1,d)`.  The fixed branch `pi(b+1)=b+1` is routed through the same
branch: fixedness gives `S(b+1,c)`, and denying `S(b+1,d)` creates
a length-four incidence path from `S(b)` to `S(d)` that contradicts the
direct length-two path from `pi(b,d)`.  The last branch `S(b+1,d)` is now
closed: if `S(c,d)` failed, incidence-path uniqueness forces `pi(b+1,c)`;
then the shifted left link gives `S(a,b)`, and the incidence-square
obstruction contradicts the branch.  The
active endpoint signature therefore contains no `hRightS_b_c` or
`hRightT_b1_c` or `hRightFixed_b1` or `hRightS_b1_d` parameter.
The left start-adjacent slice `c-a=3, b=a+1` is fully discharged at the active
endpoint.  The left-start `S(b,c)` usage in that now-discharged slice is
supplied through the global right-frontier wrapper; this is distinct from the
right gap-three special branch `S(b,c)` above, which is now discharged locally.
The fixed `a+2` branch and the `pi(b,a+2)` branch both force the left-start
`S(b,c)` contradiction.  The residual `not pi(a,a+2)` and `S(a,a+2)` branch is
closed by copying the contour step from `a+2` to `c` into a length-two
incidence path from `S(a)` to `S(c)`; tree uniqueness compares it with the
direct path from `pi(a,c)` and forces the forbidden link `pi(a,a+2)`.
The left gap-three end-adjacent slice `c-a=3, b=a+2` is reduced to two finite
branches.  The branch `pi(a,a+1)` is absorbed by the shifted gap-two left
theorem.  The branch `pi(a+1,b)` is absorbed by the already visible
start-adjacent branch on the shifted crossing `(a,a+1,c,d)`.  The
adjacent-repeat branch `S(a+1,b)` is normalized to the fixed-point branch
`pi(a+1)=a+1`; the other theorem-facing end-adjacent branch left in that
slice is `S(a+1,c)`.

These branch counts are endpoint-local snapshots, not a progress percentage.
Every finite-branch removal must still restate the non-local leaves separately:
the large-gap Biane inputs, the nested non-direct return-interval collision
theorem, and the Chapuy/trisection recurrence. A wrapper only counts as real progress when the
named endpoint exposes strictly fewer branch hypotheses after the change.

## Verification tiers

Use three tiers, in this order:

1. micro-check a local theorem or signature with `lake env lean`;
2. quick-check the enclosing module with `scripts/lean_quick_check.py`;
3. rebuild the enclosing module with `lake build`.

Only after the module build passes should the ledger, board, article note, or
blueprint be updated.

## Branch-probe discipline

The Appendix B file is large and declaration order matters.  New Biane finite
branch lemmas should therefore be developed outside the endpoint first:

1. test the proposed branch implication in small concrete models;
2. prove the helper in a downstream Lean probe importing
   `PptFactorization.AppendixBAubrunGraduate`;
3. only then move the checked helper into the earlier local branch cluster, so
   the public endpoint never references a declaration that appears later in
   the file.

The finite sanity command is:

```text
python3 scripts/aubrun_branch_sanity.py --max-n 9
```

This script enumerates small left-tight permutations and searches for
counterexamples to the currently exposed branch implications.  Since U-AUB-322
closed the finite gap-three branch list, it now also reports a live large-gap
profile.  Through `n=9`, the profile checks 6,916 left-tight permutations and
finds no crossing hypotheses at all.  This is not proof evidence: it supports
the expected direct noncrossing/large-gap route, but it does not close the
structural Biane theorem.  Its purpose is only to stop obviously false local
branch attempts before they enter Lean.

## Blueprint track

Leanblueprint has been initialized in `blueprint/`.

Use these conventions:

1. Put mathematical prose in `blueprint/src/content.tex`.
2. Attach Lean declarations using `\lean{...}`.
3. Mark statements with `\leanok` only after the declaration builds and the
   axiom audit is benign.
4. Keep open theorem-strength leaves visibly unmarked or marked as open in
   prose.
5. For conditional reduction theorems, say explicitly that `\leanok` verifies
   the reduction from its hypotheses, not the hypotheses themselves.
6. Every open leaf should name the missing invariant it is expected to prove
   or preserve, not only the Lean placeholder name.

Useful commands once content is stable:

```text
CFLAGS="-I/opt/homebrew/include" LDFLAGS="-L/opt/homebrew/lib" \
  uvx leanblueprint pdf
```

Then generate the web declaration cache:

```text
CFLAGS="-I/opt/homebrew/include" LDFLAGS="-L/opt/homebrew/lib" \
  uvx leanblueprint web
```

Finally check that every `\lean{...}` declaration referenced from the
blueprint exists:

```text
CFLAGS="-I/opt/homebrew/include" LDFLAGS="-L/opt/homebrew/lib" \
  uvx leanblueprint checkdecls
```

In this repo, `lake build EnsX2026` may be needed once before `checkdecls`,
because the Lake project still advertises that auxiliary library.

Doc generation through `doc-gen4` is deferred.  The first attempt tried to
switch to `leanprover/lean4:v4.31.0-rc1` and download a large toolchain.  Do
not enable that until the project deliberately chooses a documentation
toolchain policy.

## Adversarial review loop

After any significant proof-frontier or exposition change:

1. ask Grok for adversarial feedback on both the article-style exposition and
   the workflow note;
2. treat the feedback as advisory, not as proof evidence;
3. patch only claims that can be verified against Lean declarations, build
   output, axiom audits, or generated artifacts;
4. rerun Pandoc and leanblueprint checks after incorporating accepted changes.

Use the project review command:

```text
tools/grok_review_u_aub_02.sh
```

It asks for adversarial feedback from the current theorem-frontier context and
keeps the retained Markdown review at `build/grok_u_aub_02_review.md`
mathematical.

Durable summaries of useful Grok feedback should be tracked rather than left
only under `build/`.  The current summary is the Grok workflow review note in
the exposition folder.  Its main guardrail is to
foreground the four open theorem-strength leaves before the long checked
endpoint name, because a checked conditional adapter can otherwise feel
psychologically close to a done ticket.

Before updating this workflow note or the article note, print the current
theorem header of the sharp `AubrunBiDefectCountBound_of_*` declaration and
confirm that the listed leaves match its parameters: large-gap left outside
`c-a=3`, large-gap right outside `d-b=3` with `c=b+2`, no finite gap-three
branch parameters, the five unchanged side-specific residual collision
inputs, the smaller nonwrapping child internal-subinterval input, and `hstep`.

The recurring review questions are:

1. Does the exposition overclaim a conditional reduction as an unconditional
   theorem?
2. Are all theorem-facing leaves named in plain mathematical terms?
3. Does every `\leanok` marker correspond to a declaration that builds and has
   a benign axiom audit?
4. Are the remaining invariants structural, or is the text merely listing
   cases without a proof strategy?
5. Does the endpoint-local branch ledger strictly shrink, while the hard
   collision and Chapuy leaves remain visible when they are still open?
